# Data Processing Tools

This repository contains tools to transform raw **concordance tables (CSV)** into a **normalized JSON dataset** suitable for relational database ingestion.

## Project Structure

```
datatooling/
├── data/
│   ├── raw/           # INPUT: Original CSVs
│   └── processed/     # WORKSPACE: Generated by the script
│       ├── json/      # Validated .json files ready for DB upload
│       └── reports/   # \_audit.md & \_FIX_ME.csv
├── schemas/           # RULES: JSON Schema for data validation
├── scripts/           # LOGIC: Python modules
└── README.md
```

## Getting Started

### Quick start

1. Place your CSV in `data/raw/`.

2. Run the script:

```bash
   # If your file is named 'data.csv', no arguments are needed:
   python3 scripts/csv_to_json.py

   # For any other filename:
   python3 scripts/csv_to_json.py data/raw/your_filename.csv
```

3. Check `data/processed/json/` for the result!

> **Tip:** Naming your main working file `data.csv` saves you from typing the path every time.

---

To verify the parser logic, you can run the transformation on datasets(`normal_data.csv`, `edge_cases.csv`, and `error_cases.csv`).

### 1. Process Normal Data

This dataset contains clean, one-to-one mappings between editions.

```bash
python3 scripts/csv_to_json.py data/raw/normal_data.csv
```

### 2. Process Edge Cases

This dataset stress-tests the parser against irregularities like ranges, suffixes, and non-numeric references.

```bash
python3 scripts/csv_to_json.py data/raw/edge_cases.csv
```

### 3. Process Error Cases

This dataset is designed to fail. It contains intentional typos and schema violations (e.g., "one" instead of `1`, or "bliss" instead of `bis`) to demonstrate the "Fix-It" workflow.

```bash
python3 scripts/csv_to_json.py data/raw/error_cases.csv
```

## Validation & "Fix-It" Workflow

The tool enforces a "Safe-Fail" pipeline. If a CSV contains data that violates the schema (e.g., "bliss" instead of "bis"), the script will **abort** the JSON creation and generate a diagnostic report.

1. **Identify Errors**: If validation fails, go to `data/processed/reports/`.

2. **Open the Fix-It File**: Find `[filename]_FIX_ME.csv`.

3. **Cross-Reference**: Use the `Source_CSV_Row` column to find the exact line in your original CSV.

4. **Action**: Fix the issues in your **original raw CSV** based on the `Error_Description` column and run the script again.

## Audit & Quality Assurance

When validation passes, a human-readable **Audit Report** (`_audit.md`) is generated in the `data/processed/reports/` folder.

> **Note**: The audit report tracks **irregularities**, not necessarily errors. These are "points of interest" that may be perfectly valid for the dataset but should be manually verified to ensure no data was missed during entry.

The report categorizes the following observations:

- **Sellier Missing**: Fragments omitted by Sellier but present in other editions.  
  _> Check if these should indeed be "orphaned" references._

- **Incomplete Editions**: Rows missing one or more of the B, L, or LG references.  
  _> Check if a reference was accidentally left blank._

- **Ranges/Multi-refs**: Successful expansion of entries like `292-293` or `835, 192`.  
  _> Verify the numbers are split correctly._

- **Suffixes**: Detection of approved modifiers (`bis` or `ter`).  
  _> Ensure the suffix belongs to that specific reference._

- **Non-Numeric**: Identification of references like `app. XIII`.  
  _> Confirm these shouldn't have been converted to digits._
